## 一、研究动机与问题

### 1.1 背景

OneRec 系列生成式推荐模型已广泛应用于快手短视频、直播、广告和电商等多个业务场景。这些模型受益于 **Scaling** 效应，但其 **推理（Reasoning）** 能力始终未能被激活——因为仅由 item token 组成的序列无法构建有意义的思维链（Chain-of-Thought, CoT）。

### 1.2 核心发现

受 LLM 领域"先思考后回答"（think-before-answer）范式的启发，OneRec-Think 和 OpenOneRec 进行了初步探索，成功将思考模式泛化到了推荐任务中。然而发现了一个**意外现象**：

> **思考模式（thinking mode）相比非思考模式（non-thinking mode）在推荐基准上没有展现出明显优势。**

### 1.3 根因分析

借鉴多模态大语言模型（MLLM）的最新研究，论文认为推荐系统有效推理依赖于两个关键因素：

| 维度 | 含义 | 失败表现 |
|------|------|----------|
| **感知 (Perception)** | 将 item token 与其背后的语言语义相关联的能力 | 模型可能"幻觉"出错误的 item 内容 |
| **认知 (Cognition)** | 将用户行为序列重组为连贯的潜在兴趣点的能力 | 逻辑混乱、无法正确推断用户意图 |

## 二、OneReason 技术方案

### 2.1 整体架构

OneReason 包含三个核心训练阶段：

```
预训练（四粒度对齐）→ SFT（三级认知CoT）→ RL（先专后统）
```

### 2.2 预训练阶段 —— 四粒度数据体系

为解决 item token 与自然语言的对齐问题，构建了递进式的四层数据：

| 粒度 | 内容 | 作用 |
|------|------|------|
| **Token粒度** | 子 token 组合语义预测、前缀语义预测 | 最细粒度的语义对齐 |
| **Item粒度** | 容量感知的 caption 粗粒化、多视角 Item QA | 完整 item 级别的跨模态融合 |
| **Relational粒度** | item 间直接关联 + 多 item 兴趣流 | 将协同信号显式编码到文本空间 |
| **User粒度** | 域分组行为序列 + 时间交错行为序列 | 用户级别的时序建模与跨域对齐 |

消融实验证明，四层粒度按序添加，在 R0（感知）→ R1（推导）→ R2（演化）→ R3（推荐）各层级上均带来一致性的性能提升。

### 2.3 SFT 阶段 —— 三级认知CoT

SFT 阶段围绕 R0-R3 层次构建 **压缩 + 推理** 的双轨训练：

| 层级 | 能力 | 数据构造 |
|------|------|----------|
| **R0: 感知** | item token ↔ 自然语言双向映射 + QA | 约 941K 样本（682K caption + 259K QA） |
| **R1: 推导** | 从单个 item 语义推导 item-item 关系 | 约 400K 样本（基于 TagNext CF + after-play-search） |
| **R2: 演化** | 用户兴趣随时间演化的建模 | 约 130K 样本（Action Selection / Topic Gen / Direct Gen） |
| **R3: 推荐** | 综合压缩+推理的最终推荐决策 | 三阶段 CoT 协议 |

#### R3 推荐CoT 三阶段协议

1. **Persona Abstraction（画像抽象）**：将稀疏行为压缩为类型化的用户画像先验
2. **Interest Expansion（兴趣扩展）**：从压缩状态展开多个证据支撑的候选方向
3. **Transition Inference（过渡推断）**：比较候选方向，选择最可能的下一步兴趣

**关键发现**：兴趣扩展宽度 $n \in \{1,3,5\}$ 始终优于 $n \in \{10,20\}$——候选方向过多反而稀释最强信号。

CoT 质量通过五个维度评估：**安全性**（无答案泄露）、**一致性**（与目标对齐）、**逻辑性**（可泛化推理）、**事实性**（忠实于用户历史）、**信息量**（提供可操作的推荐指导）。

### 2.4 RL 阶段 —— "先专后统"

#### 2.4.1 推荐导向的 GRPO

针对推荐场景的低命中率问题，设计了：

- **两阶段 Rollout**：$N$ 条 CoT 轨迹 × 每轨迹 $K$ 个 item token 序列 = $N \times K$ 个有效 rollout，仅需生成 $N$ 条推理轨迹
- **多样性奖励**：$R_{\text{div}}$ 基于首个子 token 的类别多样性
- **分级裁剪**：CoT token 使用宽松裁剪范围，item token 使用紧缩裁剪范围
- **负样本降权**：未命中样本权重 $\beta < 1$，缓解稀疏奖励下的梯度稀释

#### 2.4.2 统一策略：RFT vs MOPD

| 方法 | 原理 | 特点 |
|------|------|------|
| **RFT (拒绝采样微调)** | 离线收集各域教师的"黄金"推理轨迹，再训练统一模型 | 稳定、thinking 模式始终优于 non-thinking |
| **MOPD (多教师在线蒸馏)** | 在线蒸馏多个域专家策略，配合信息增益感知的轨迹过滤 | Product/Live 域更优，thinking/non-thinking 同步提升 |

**核心结论**：RFT 和 MOPD 各有所长，均优于混合域 RL。但 MOPD 受教师上界限制，学生很难大幅超越教师。

## 三、OneReason-Bench 基准

构建了递进式推理评估基准：

| 层级 | 任务类型 | 评估指标 |
|------|----------|----------|
| **R0: 感知** | Item Understanding, Itemic Pattern Grounding, Item QA | LLM-as-a-Judge, Pass@K, Accuracy |
| **R1: 推导** | Item2Item（从源 item 选择最相关的目标 item） | Accuracy |
| **R2: 演化** | Evolution Action Selection, Topic Gen, Direct Gen | F1, Action-Logic Score |
| **R3: 推荐** | 单域/跨域推荐（Video / Product / Ad / Live） | Pass@K, Recall@K |

## 四、实验结果

### 4.1 推荐性能

OneReason RFT thinking 在跨域推荐四个子任务上均取得 **SOTA**：

| 方法 | Cross-Video Pass@64 | Cross-Product Pass@64 | Cross-Ad Pass@64 | Cross-Live Pass@64 |
|------|--------------------|-----------------------|-------------------|---------------------|
| GPT-5.4 | 0.24 | 1.43 | 1.64 | 7.20 |
| LC-Rec-PT-SFT-8B | 1.49 | 3.95 | 15.85 | 19.32 |
| **OneReason RFT thinking** | **2.41** | **5.47** | **17.78** | **21.10** |

### 4.2 推理优势 —— 首次实现 thinking > non-thinking

OneReason 是**首个**在推荐基准上 thinking 模式持续优于 non-thinking 模式的模型：
- SFT 阶段：thinking 反而不如 non-thinking（负 $\Delta$LL）
- RFT 阶段：thinking 在所有域上全面超越 non-thinking（正 $\Delta$LL）

CoT 分析指标（4 维度诊断框架）：

| 指标 | SFT | RFT |
|------|-----|-----|
| $\Delta$LL（CoT 似然增益） | 全负（CoT 分散注意力） | 全正（CoT 提升目标似然） |
| $\ell_t$ 进展 | CoT 前缀逐步改善，但收敛慢 | 更早饱和，仅需少量前缀即可 |
| $\gamma_{\text{legal}}$ | 100%（无非法 item 引用） | 100% |
| $\gamma_{\text{hist}}$ | 较高 | Video/Ad 更严格 |

### 4.3 通用能力保持

OneReason 保留了 Qwen3-8B 基座的通用推理能力（MMLU-Pro 72.08%、MATH-500 95.40%），而 LC-Rec 变体出现严重退化。

### 4.4 有趣发现：CoT 监督可提升 non-thinking 推理

在相同训练 token 预算下，用部分 CoT 数据替换 unCoT 数据，可在 Video、Product、Live 三个域上提升 **non-thinking** 推理性能。CoT/unCoT 存在域依赖的最优混合比——Ad 域偏向 unCoT-heavy。这暗示 CoT 监督中的压缩信号可能迁移到了直接解码。

## 九、深度 Q&A：训练数据全貌

### Q1: RL 阶段有特定的数据构造吗？

**回答**：RL 阶段的"数据"主要是两种形态——**重用的 prompt** 和**模型自生成的轨迹**。具体分述如下：

#### 一、Phase 1（Domain-Specific GRPO）：不构造新数据，重用已有 Prompt

GRPO 阶段**没有构造新的训练 prompt 或标注**。输入数据直接复用的是：

- **域特定 GRPO**：对 Video / Product / Ad / Live 四个域分别训练，输入的 prompt 就是 **R3 推荐任务的标准格式**——用户画像 $\mathcal{P}$ + 多域交互历史 $\mathcal{H}$，要求模型预测下一个 item
- **Mix-RL**（baseline）：论文明确提到是"applying GRPO to mixed data covering **R2 evolution and R3 recommendation tasks** from all domains"——就是把 SFT 阶段已有的 R2（兴趣演化）和 R3（推荐）任务的 prompt 直接拿来用

GRPO 的"数据流"本质是**模型自己生成轨迹作为训练信号**——对于每个用户 $u$，模型采样 $N \times K$ 个 rollout，然后基于 ground-truth item 命中与否计算 reward。**不需要人工构造任何新数据。**

#### 二、RFT：构造了 $\mathcal{D}_{\text{RFT}}$（唯一显式数据构造）

RFT 是 RL 阶段**唯一从零构造训练数据**的环节：

**构造流程**：
```
1. 四个域特定 GRPO teacher → 对每个用户各自采样多条 trajectory
2. 筛选：保留 predicted item ∈ ground-truth set C⁺ᵤ 的轨迹
3. 质量过滤：去掉低质量/不一致的 CoT（沿用 SFT 阶段的五维评估体系）
4. 跨域聚合：合并四个域的所有"黄金轨迹"
5. 产出：D_RFT = {(x_u, CoT_u, c_u)}
```

**数据特点**：

| 维度 | 说明 |
|------|------|
| 数据来源 | 模型**自生成**（teacher 采样），非人工标注 |
| 筛选条件 | **仅保留命中 + CoT 高质量**的轨迹 |
| 数据格式 | 与 SFT 的 R3 CoT 格式完全相同：用户 context → CoT → itemic tokens |
| 训练方式 | 标准的 next-token prediction（与 SFT 相同），不是 RL |
| 核心价值 | 将 teacher 通过 self-exploration 发现的"超越 SFT teacher"的推理策略固化为监督信号 |

**关键洞察**：$\mathcal{D}_{\text{RFT}}$ 中的 CoT 不是 SFT 阶段由 teacher LLM 生成的，而是**经过 GRPO self-exploration 后强化过的模型自己生成的**——这些推理路径比原始 SFT teacher 更强，因为经过了 reward 信号的筛选和优化。

#### 三、MOPD：不构造数据集，使用 On-Policy 动态采样

MOPD 完全**不构造静态数据集**：

- **输入 prompt $x$**：同样复用 R2+R3 任务的标准 prompt，来自 SFT 阶段的训练数据
- **训练轨迹 $y$**：由 student 模型**实时采样**（on-policy），每步都是新鲜的
- **信息增益过滤**：不是数据预筛选，而是训练时**动态丢弃**低信息轨迹（$|\hat{A}| \approx 0$ 的已对齐样本），保留高差异样本进入当前 batch 的 loss 计算

> "For a given prompt $x$, we first sample a complete rollout $y = (\mathbf{CoT}; \mathbf{c}) \sim \mu_\theta (\cdot \mid x)$."

#### 四、总结对比

| 维度 | SFT | GRPO | RFT | MOPD |
|------|-----|------|-----|------|
| **输入 prompt 来源** | Teacher LLM 构造 | 复用 SFT 的 R2/R3 prompt | 复用 SFT 的 R2/R3 prompt | 复用 SFT 的 R2/R3 prompt |
| **输出 target 来源** | Teacher LLM 生成 CoT | 模型自采样（rollout） | Teacher 模型采样 → 筛选 | Student 模型实时采样 |
| **是否构造新数据集** | ✅ 构造 6 组训练数据 | ❌ 不构造 | ✅ **构造 $\mathcal{D}_{\text{RFT}}$** | ❌ 不构造 |
| **数据筛选** | 五维质量评估淘汰 | Reward 信号自动筛选（命中/未命中） | **仅保留命中 + 高质量 CoT** | 信息增益感知动态过滤 |
| **人工标注需求** | 有（teacher LLM + 人工 QA） | 无 | 无（自动筛选） | 无 |

#### 五、核心结论

**RL 阶段的"数据构造"只有一个**：RFT 的 $\mathcal{D}_{\text{RFT}}$。但它与 SFT 的数据构造有本质区别：

- **SFT 数据** = 外部 teacher 生成 → 人工质量把控 → 静态数据集
- **RFT 数据** = 模型自我探索生成 → 自动根据 ground-truth 筛选 → 对更强推理策略的"提取固化"

GRPO 和 MOPD 都**不需要构造新数据**——它们的力量来源是 RL 训练范式本身（通过 reward/teacher 信号优化策略），而非更高质量的训练数据。这也正是 RL 区别于 SFT 的核心：**从"喂更好的数据"转向"让模型自己探索更好的策略"**。

### Q2: 所有这些训练数据是如何产生的呢？

**回答**：OneReason 的训练数据横跨预训练、SFT、RL 三个阶段。按**数据来源**可以归为四类：业务真实日志、LLM 生成、公开数据集、模型自生成。

---

#### 一、预训练数据（~578B tokens）

##### 1. Token Granularity（2.50%）

| 子数据集 | 如何产生 |
|----------|---------|
| Single-Token Semantic Prediction | 对单个 sub-token `<a_xxxx>`，找共享该 token 的所有 item，**LLM 生成**该 sub-token 的语义描述 |
| Compositional Prefix Semantic Prediction | 对 prefix 对 `<a_xxxx><b_xxxx>`，找共享该 prefix 的 items，**LLM 总结**组合语义 |
| Prefix Itemic Token Grounding | 上一条的逆方向，给定文本描述 → **LLM 生成** 让模型选对应 prefix |
| Part-to-Whole Semantic Prediction | 强制两步推理：**LLM 生成**先分 token 解释、再合成整体 caption |

> 所有 token 粒度的数据**来源**是业务 item 库（真实 item 的聚类），**生成工具**是 LLM。

##### 2. Item Granularity（31.84%）

| 子数据集 | 如何产生 |
|----------|---------|
| Item Captioning | **业务真实 item metadata**（视频封面/帧、商品描述、直播截图、广告素材）→ **多模态 LLM 生成** dense caption → **规则粗粒度化**（Capacity-Aware Coarse-Graining：去除 OCR 噪声、连续值分桶、保留品类/IP/卖点骨架） |
| Item Grounding | 上一条的逆方向，文本描述 → itemic token |
| Item QA | 从验证后的 captions + item metadata 出发，**LLM 生成**多视角 QA（受众偏好、核心卖点、视觉风格、负反馈原因等） |

> item 数据来自真实业务物料，LLM 做 caption 生成和 QA 扩充。

##### 3. Relational Granularity（3.18%）

| 子数据集 | 如何产生 |
|----------|---------|
| Direct Item-to-Item | **业务 search-after-play 日志**（用户看视频后搜索并购买商品）→ 抽取视频-商品 pair → **模板融合**双方 caption 生成自然语言解释 |
| Multi-Item Interest Flows | **三个来源**：search-after-play 表 + TagNext CF（协同过滤视频对）+ **用户行为序列滑动窗口共现** → 构建全局 item 图 → 随机采样长链 → 取间隔子链（去语义太近的）→ **LLM 生成**每个跳转兴趣转移解释 |

> 关系数据的原材料来自真实的用户协同行为日志，LLM 负责将隐式的 item 转移转化为显式的自然语言解释。

##### 4. User Granularity（34.05%）

| 子数据集 | 如何产生 |
|----------|---------|
| Domain-Grouped Behavior | **业务真实用户行为日志** → 按域分组 itemic tokens → **LLM 清洗**用户画像（去噪、去模板化 → 流畅自然语言叙事）→ 构造**多轮 QA 对话**格式 |
| Chronologically Interleaved | 相同用户行为，但**严格按时间戳排列**跨域混合 → **概率性替换**部分 itemic token 为其文本 caption → 形成文本-token 交错序列 |

> 用户数据的原材料就是用户的真实画像 + 跨域交互历史，LLM 只做清洗和格式结构化。

##### 5. General-Domain Corpora（28.44%）

| 类别 | 数据来源 |
|------|---------|
| 通用文本（26.85%）| **公开数据集**（Nemotron-CC-Math、OpenMathReasoning、Code、MMLU 等 20+ 个） + 业务内部文本 |
| 多模态（1.59%） | **公开数据集**（UNO-1M、BLIP3o、AnyEdit 等）+ 业务内部图文交错数据，图片先 tokenize 为 itemic pattern |

---

#### 二、SFT 数据（~395 万条）

##### R0：Perception（94.1万，23.82%）

| 子数据集 | 如何产生 | 与预训练的区别 |
|----------|---------|:---:|
| 四域 Caption（68.2万）| **复用预训练 item pool** → **LLM 生成 CoT 格式**：先解释三层 token 语义如何逐级支撑最终 caption → 再给 caption；同时保留 **UnCoT 版本**（直接 caption） | 预训练是 `token→caption`，SFT 是 `token→CoT（分层解释）→caption` |
| Video / Product QA（25.9万）| 同上 item pool → **LLM 生成**面向属性、受众、负反馈等维度的问答 | 预训练 QA 基础，SFT 扩充更多视角 |

##### R1：Derivation（40万，10.12%）

| 子数据集 | 如何产生 | 关键筛选 |
|----------|---------|---------|
| TagNext CF（~35.8万 pairs）| **TagNext 协同过滤信号** → **本地部署旗舰 LLM**对每个 pair 做 explicit-relevance 三分类判断（显式相关/显式无关/不确定）→ **仅保留"显式相关"** | 去掉仅靠人群 overlap/流行度的弱关联 |
| After-Play-Search（~38.8万 pairs）| **search-after-play 表** → 同上 LLM 判断 + 筛选 → 保留后 → **LLM 提取 bridge 变量**（源需求、桥类型、抽象桥、延续方向、原因种子、置信度）→ **LLM 仅基于源侧证据+bridge 变量生成解释**（目标 item 不暴露给解释生成步骤） | 防泄漏：目标 item 仅用于一致性检查 |

##### R2：Evolution（13万，3.28%）

| 子数据集 | 如何产生 |
|----------|---------|
| Multi-Hop Reasoning | **全域用户行为时间线**（真实日志）→ **强 LLM 识别关键演化节点**（有意义的用户意图偏移/细化/关闭）→ 提取 candidate evolution chains → **另一个 LLM 做 Judge 质量审查**（检查时序敏感性、认知增量、因果排他性等 11 项）→ 淘汰伪逻辑链、随机消费链、弱基础转移 → 构造三种任务格式（Action Selection / Topic Generation / Direct Generation）|

##### R3：Recommendation（88.4万，22.36%）

| 子数据集 | 如何产生 |
|----------|---------|
| CoT 推荐数据（29.6万）| 三要素输入：**用户画像**（真实）+ **交互时间线**（真实）+ **ground-truth 下一次交互**（真实日志）→ **Teacher LLM** 执行三阶段 CoT 协议：**Persona Abstraction** → **Interest Expansion**（$n \in \{1,3,5\}$ 宽度控制）→ **Transition Inference** → **后处理**（将历史 item ID 替换为 itemic tokens，删除匹配失败的引用）→ **五维质量评估**（Safety / Consistency / Logic / Factuality / Informativeness）→ 低分 trace 淘汰 |
| UnCoT 推荐数据（58.8万）| 同样输入 → 直接输出 target itemic tokens，无 thinking trace |

##### Itemic Instruction（10.3万，2.61%）

| 如何产生 |
|---------|
| 预定义 **task tag 清单**（涵盖转换/检索/匹配/比较/生成/指令控制 6 大类）→ **LLM 为每个 tag 生成**合适的指令 + 采样用户侧物料配对 → 规则一致性检查（token 引用与素材匹配） |

##### General-Domain（150万，37.96%）

| 如何产生 |
|---------|
| 直接使用公开的 **Step-3.5-Flash-SFT 数据集** → 保守预处理（去掉 tool-use 样本、只保留 system/user/assistant 三角色格式、去掉格式损坏样本） |

#### 三、RL 数据

| 阶段 | 数据来源 | 是否构造新数据 |
|------|---------|:---:|
| **Domain-Specific GRPO** | 直接复用 SFT 的 **R2 演化 + R3 推荐任务 prompt**，输入格式完全一致 | ❌ |
| **Mix-RL** | 同 GRPO，四域 R2+R3 混合 | ❌ |
| **RFT** | 四个域 expert teacher 各自采样 rollout → **仅保留命中 ground-truth 的轨迹** → 质量过滤 → 跨域聚合为 $\mathcal{D}_{\text{RFT}}$ | ✅ **唯一新构造的数据集** |
| **MOPD** | 复用 R2+R3 prompt + **student 实时 on-policy 采样** → 信息增益过滤动态丢弃低差异轨迹 | ❌ |

#### 四、总览：数据产出的"分工"

```
                    ┌── 业务真实日志 ───────────────────────────┐
                    │  • 用户行为序列（user/item granularity）      │
                    │  • search-after-play（relational）            │
                    │  • TagNext CF（relational + R1）              │
                    │  • 用户画像 + ground-truth（R3）              │
                    │  • item metadata/images（item granularity）   │
                    └────────────────────────────────────────────┘
                                       │
                    ┌── LLM 生成 ─────────────────────────────┐
                    │  • Token 语义描述（token granularity）        │
                    │  • Multi-perspective QA（item granularity）   │
                    │  • 兴趣转移解释（relational granularity）     │
                    │  • 演化链提取 + Judge（R2）                  │
                    │  • 三阶段 CoT trace（R3）                    │
                    │  • 六组 itemic instruction 任务              │
                    │  • 用户画像清洗（user granularity）           │
                    └────────────────────────────────────────────┘
                                       │
                    ┌── 公开数据集 ───────────────────────────┐
                    │  • 通用数学/代码/推理/医学（预训练）          │
                    │  • 多模态图文（预训练）                      │
                    │  • Step-3.5-Flash-SFT（SFT 保底）           │
                    └────────────────────────────────────────────┘
                                       │
                    ┌── 模型自生成 ───────────────────────────┐
                    │  • D_RFT（teacher GRPO 采样 + filter）       │
                    │  • MOPD 实时 rollout（student on-policy）    │
                    └────────────────────────────────────────────┘
```

**核心规律**：越靠近底层的阶段（预训练），数据越依赖**真实业务日志**做 Modality Alignment；越靠近上层（SFT/RL），**LLM 的生成和筛选**占比越大，用来构造推理能力和 coaching 信号。RL 阶段几乎不再引入外部数据，完全依赖模型**自我探索**产生优化信号。

## 十、PT 与 SFT 任务详解与数据示例

### 10.1 预训练四粒度任务详解

#### 10.1.1 Token Granularity（Token 粒度）

**设计动机**：传统方法将整个 itemic pattern 对齐到整体 caption，忽略子 token 间的语义组合过程。Token 粒度数据显式建模子 token 逐级组合成完整 item 语义的过程。

**包含 4 种子任务**：

1. **Single-Token Semantic Prediction**：单个 sub-token → 语义描述
2. **Compositional Prefix Semantic Prediction**：前缀 sub-token 对 `<a_xxxx><b_xxxx>` → 其组合语义
3. **Prefix Itemic Token Grounding**：文本描述 → 对应的 prefix sub-token 对（上一条的逆方向）
4. **Part-to-Whole Semantic Prediction**：强制两步推理：先逐 token 解释 → 再合成整体 caption

**数据示例（Compositional Prefix Semantic Prediction）**：

```
User: 短视频域中，<a_3664><b_3076> 表示什么？
Assistant:
这些短视频在内容主题上高度聚焦于"都市夜景中的个体孤独与情感内省"。画面通常选取夜
晚的城市街道、霓虹灯下或夜市背景，核心视觉元素为背对镜头的单人（多为背影），通过人
物与繁华或静谧环境的对比，营造一种疏离、沉思或略带忧郁的氛围。视频往往借助文字叠加
来外化人物内心的挣扎、对过去的追忆、对情感的执着或对生活的感悟，将物理空间的"夜"转
化为心理空间的"独白"。
从受众角度来看，这类内容主要吸引那些在快节奏都市生活中感到孤独、压力或情感共鸣的观
众。目标人群倾向于通过观看此类视频寻找情感慰藉、确认自身情绪，或在静谧的视觉与文字
中寻求片刻的内心平静与精神寄托。观众画像多为年轻至中年群体，对情感类、治愈系或略带
伤感氛围的都市生活记录有较高偏好。
```

**Token Ratio**: 2.50%（四粒度中占比最小）

---

#### 10.1.2 Item Granularity（Item 粒度）

**设计动机**：将 itemic pattern 与自然语言描述对齐。相比 OpenOneRec 的原始做法，引入两大关键增强：
1. **Capacity-Aware Caption Coarse-Graining**：防止过于细粒度的 caption 与仅 3 个 sub-token 的表示能力不匹配导致幻觉
2. **Multi-Perspective Item QA**：从目标受众、核心卖点、视觉风格、负反馈原因等多角度深化语义对齐

**包含 3 种子任务**：

| 子任务 | Token Ratio |
|--------|------------|
| Item Captioning | 17.28% |
| Item Grounding（文本→token） | 12.12% |
| Item QA | 2.44% |

**数据示例（Item Captioning，容量感知粗粒度化后）**：

```
User: 请描述一下这个电商商品 <|prod_begin|><a_2147><b_7978><c_5031>。
Assistant:
该商品是一款户外软壳冲锋衣，一级类目是运动户外鞋服，二级类目是户外服装，三级类目是
软壳衣裤。采用连帽拉链设计，软壳材质兼具防风防水和加绒保暖功能，标准版型，加绒内里，
多色可选，简约时尚，适合户外活动及日常休闲。价格区间为100-120元。无品牌。
```

> **注意到**：caption 已经过粗粒度化——精确价格被映射到区间，无品牌信息，保留了品类/功能/适用场景骨架。

**Token Ratio**: 31.84%（四粒度中第二大）

---

#### 10.1.3 Relational Granularity（关系粒度）

**设计动机**：将隐式的协同过滤信号显式编码为自然语言，使 item 间转移具有可解释的语义基础。

**数据格式**（交替序列）：

\[
\text{Itemic\_Pattern}_0 \rightarrow \text{Textual\_Explanation}_0 \rightarrow \text{Itemic\_Pattern}_1 \rightarrow \cdots
\]

**包含 2 种子任务**：

| 子任务 | 数据来源 |
|--------|---------|
| Direct Item-to-Item Associations | search-after-play 表（用户看完视频→搜索→购买商品） |
| Multi-Item Interest Flows | search-after-play + TagNext CF + 滑动窗口共现 |

**数据示例（Multi-Item Interest Flow，四跳）**：


> **多跳关联**：直播健身主播 → 腿部训练视频 → 普拉提圈 → 大腿抗阻训练器。每跳都有自然语言解释兴趣转移的"为什么"。

#### 10.1.4 User Granularity（User 粒度）

**设计动机**：将跨模态对齐扩展到用户级别，使模型在个性化上下文中理解和生成多模态内容。

**包含 2 种数据格式**：

| 格式 | 构造方式 |
|------|---------|
| Domain-Grouped Behavior Sequences | 文本用户画像 + 按域分组的 itemic tokens，通过**多轮 QA 对话**动态结构化 |
| Chronologically Interleaved Behavior Sequence | 严格按时间戳排列、跨域混合；**概率性替换**部分 token 为文本 caption |

**数据示例（Domain-Grouped 多轮 QA）**：

```
Assistant: ['<|ad_begin|><a_7586><b_7762><c_7321>',
           '<|ad_begin|><a_1512><b_2007><c_5218>']
```

### 10.2 SFT 各层级任务详解

#### 10.2.1 R0：Perception（感知层）

**目标**：建立 itemic token 与自然语言之间的双向映射能力，并具备基于 token 回答内容问题的能力。

**包含任务格式**：

| 任务 | 样本量 | 格式 |
|------|--------|------|
| CoT Token-to-Caption | 68.2万 | `<think>`分层解释三层 token → 整体 caption `</think>` caption |
| UnCoT Token-to-Caption | — | token 直接 → caption |
| Caption-to-Token Grounding | — | 文本描述 → itemic token |
| Product/Video QA | 25.9万 | 给定 token + 问题 → 选项/答案 |

**数据示例1：CoT Itemic-Token-to-Caption Generation**

```
System: 你是一名专业的视频内容理解与描述助手，请根据短视频 itemic tokens 生成准确
        的视频描述。
User: 请描述短视频 <|video_begin|><a_2344><b_3729><c_6164> 的视频内容。
Assistant:
<think>从 <a_2344> 看，该视频属于线下休闲娱乐和商场体验类内容；<b_3729> 进一步指
向新奇互动玩法，而不是普通购物或餐饮记录；<c_6164> 支持"悬挂式抓零食"这一具体场景，
核心是参与者被安全绳吊起，在零食池中抓取零食并制造轻松搞笑效果。三层 itemic tokens
的语义交集指向一种面向年轻消费者的商场趣味体验活动。</think>
视频记录了一名女性在商场内体验悬挂式抓零食游戏的过程。参与者被安全绳吊在装满零食的
区域中，通过身体动作抓取零食，旁边有工作人员协助。画面字幕强调单次体验价格和"抓零
食"的趣味性，整体呈现为轻松、搞笑、适合线下娱乐分享的短视频。
```

> **CoT 格式**：先分层解释 `<a_xxx>` / `<b_xxx>` / `<c_xxx>` 各自语义，再合成整体 caption。实现了从"部分到整体"的可解释推理。

**数据示例2：UnCoT Token-to-Caption**

```
System: 你是一位直播理解专家，擅长将主播 itemic tokens 转换为自然的直播描述。
User: 已知主播 <|living_begin|><a_522><b_6604><c_512>，请说明这位主播通常会直播什么内容。
Assistant: 该主播定位为音乐轮播类无人直播博主，以虚拟直播间为载体，主打流行歌曲、
          古风歌曲和轻量 DJ 音效的持续播放。直播中通常没有真人强互动，更多承担背景
          音和陪伴功能，适合观众在休息、睡前或碎片化时间收听。整体风格简约、节奏感
          强，主要吸引喜欢听歌放松、需要背景音陪伴的年轻用户。
```

**数据示例3：Product QA**

```
User: 商品 <|prod_begin|><a_7516><b_5044><c_7205> 是一款 700W 电动抛光机，采用全
      铜电机和无级变速设计，可用于汽车打蜡、封釉、划痕修复，也可用于皮具保养和木材

```

**数据示例4：Caption-to-Itemic-Token Grounding**

```
User: 请根据以下广告内容描述生成匹配的广告 itemic tokens：视频为生活技巧类广告，通
      过真人演示卫裤绳子的多种系法，展示打结、蝴蝶结、松紧节等做法，重点解决裤绳容
      易缩回裤腰的问题，并用限时补贴弹窗引导用户点击领取福利。

```

> R0 与预训练的本质区别：**预训练**是 `token → caption` 的直接映射，**SFT** 是 `token → CoT（分层解释三层 token 语义如何支撑 caption） → caption`。

---

#### 10.2.2 R1：Derivation（推导层）

**目标**：从单个 item 的语义推导 item-item 之间的常识性关系。

**数据构造流程（两阶段 LLM 管线）**：

```
阶段1 Explicit Relevance Judgement
  → TagNext CF pair / search-after-play pair → 本地旗舰 LLM 三分类判断
    （显式相关 / 显式无关 / 不确定）
  → 仅保留"显式相关"

阶段2 Relation Explanation Generation
  → 保留的 pair → LLM 提取 Bridge 变量
    （source_need / bridge_type / bridge / direction / reason_seed / confidence）
  → LLM 仅基于源侧证据 + bridge 变量生成解释
    （目标 item 不暴露给解释生成步骤，仅用于一致性检查）
```

**Bridge 变量提取 Prompt 模板**：

```
给定源视频信息和一组抽象桥接变量，请你以推荐分析师视角，输出一段自然流畅的第一人称
中文推理，说明用户为什么会从当前内容继续延伸到某个相邻内容方向。候选目标信息仅用于
内部校验，你不能在结果里显式提到目标。

源视频信息：
- src_pid: {src_pid}
- src_tag: {src_tag}
- src_dense_caption: {src_dense_caption}

抽象桥接变量：
- source_need: {source_need}
- bridge_type: {bridge_type}
- bridge: {bridge}
- direction: {direction}
- reason_seed: {reason_seed}

输出要求：
1. 只输出最终推理正文，正文必须以"我注意到"开头
2. 核心依据必须来自源视频信息和抽象桥接变量
```

#### 10.2.3 R2：Evolution（演化层）

**目标**：建模用户兴趣随时间的演化轨迹，识别跨时间、跨域的兴趣偏移/细化/关闭。

**数据构造流程（两阶段 LLM 判断）**：

```
阶段1 Chain Extraction
  全域用户行为时间线（真实日志）
    → 强 LLM 识别关键演化节点（有意义的意图偏移/细化/关闭）
    → 提取 candidate evolution chains
    → 11 项严格有效性准则过滤（时序互换测试、认知增量测试、因果排他性测试等）

阶段2 Quality Judgement
  候选 chain → 另一个 LLM 做 Judge 质量审查
    → 淘汰伪逻辑链、随机消费链、弱基础转移
    → 每个用户最多保留 3 条"金子般"的逻辑链
```

**R2 11 项有效性准则（核心摘录）**：

| # | 准则 | 说明 |
|---|------|------|
| 1 | 顺序互换测试 | A↔B 后逻辑仍通顺 → 非强关联，剔除 |
| 2 | 认知增量 | B 必须引入 A 未出现的新变量（参数/术语/对比维度） |
| 3 | 行为修正/收敛 | 从"发散"到"聚焦"或从"错误尝试"到"路径修正" |
| 4 | 交互深度转移 | 从被动曝光（广告/视频）→ 主动检索核心信息 |
| 5 | 时空密度 | A→B 时间跨度无相关引导行为 → 日常零散行为，非演化 |
| 6 | 非普适性 | 排除生活补货"买了就不会再买"的随机消费 |
| 7 | 触发源 | 链条必须以"主动搜索"或"带有明确意图的对比"为起点 |
| 8 | 因果排他性 | B 可以"完全独立于 A"发生 → 链条断裂 |
| 9 | 证据闭环 | logic 中引用的每个转折必须在 events 中对应 |
| 10 | 禁止脑补 | 所有逻辑转折必须由日志中的物理证据（搜索词/标题/参数）支撑 |
| 11 | 同类目平移 | 将同类目内容按时间排列并强行解释递进 → 剔除 |

**包含 3 种训练/评估任务格式**：

| 任务 | 说明 |
|------|------|
| Action Selection | 给定候选行为序列，选择支撑有效兴趣演化的行为子集 |
| Topic Generation | 给定目标兴趣主题，基于历史生成逻辑链 |
| Direct Generation | 不给定主题，直接从历史中自主发现兴趣演化链 |

#### 10.2.4 R3：Recommendation（推荐层）+ 三阶段 CoT 协议

**目标**：综合感知、推导、演化三层的压缩与推理能力，做出最终推荐决策。

**三阶段 CoT 协议**：

```
┌────────────────────────────────────────────────────────┐
│ 阶段 1: Persona Abstraction（画像抽象）                  │
│   将稀疏行为压缩为类型化的用户画像先验                      │
│   输出：候选用户画像类型 + 相比群体的特点                   │
├────────────────────────────────────────────────────────┤
│ 阶段 2: Interest Expansion（兴趣扩展）                   │
│   从压缩状态展开多个证据支撑的候选方向（宽度 n ∈ {1,3,5})  │
│   输出：A/B/C/D... 多个候选兴趣方向，各有证据               │
├────────────────────────────────────────────────────────┤
│ 阶段 3: Transition Inference（过渡推断）                  │
│   比较候选方向，权衡逻辑自洽性、证据密度和时序关系            │
│   输出：最可能的下一步兴趣 + 推荐策略                       │
└────────────────────────────────────────────────────────┘
```


#### 10.2.5 R3 CoT 五维质量评估与低分案例

对每一条 CoT trace，从五个维度评分，低分 trace 淘汰：

| 维度 | 含义 | 典型低分表现 |
|------|------|-------------|
| **Consistency** | CoT 方向是否与 target 对齐 | 虽有宠物兴趣的推理自洽，但 target 是房产，CoT 却导向鹦鹉饲养 |
| **Factuality** | CoT 不引用行为历史中不存在的证据 | CoT 中"购物中有年货窗花"实际未在用户行为中出现（幻觉） |

| **Safety** | CoT 不能泄露 target 答案 | CoT 直接写出"早泄阳痿检查"——target 答案是性健康检查广告，完全泄露 |

## 十一、在线部署架构与实践

OneReason 已在**快手本地生活广告场景**中实现稳定在线部署，在严格的延迟约束、强基线竞争和 item 可投放性限制下，取得了该场景**最显著的商业收益提升之一**。

### 11.1 核心挑战

由于 LLM 的模型规模限制，**直接部署 OneReason 做实时推荐面临严重的延迟和计算成本问题**。为此，论文提出了全新的在线部署架构——"快-慢思考"（Fast-Slow Thinking）。

---

### 11.2 "Fast-Slow Thinking" 在线架构

该架构避免完全依赖实时推理，而是引入**快管道（Online）和慢管道（Nearline）协作机制**，在保持低延迟的同时，持续将 OneReason 的知识和推理能力注入推荐系统。

```
┌─────────────────────────────────────────────────────┐
│                   Fast-Slow Thinking                  │
│                                                       │
│  Slow Pipeline (Nearline/离线)                        │
│  ┌──────────────────────────────────────────┐        │
│  │ 周期性聚合用户数据 → OneReason 离线推理     │        │
│  │ → 解码 item IDs → 写入 Redis 候选池       │        │
│  └──────────────────────────────────────────┘        │
│                       ↓ 联合送入                      │
│  Fast Pipeline (Online/在线)                          │
│  ┌──────────────────────────────────────────┐        │
│  │ OneRec 实时检索 → 与 Redis 候选池联合 →    │        │
│  │ 下游排序模型统一融合 → 最终推荐结果         │        │
│  └──────────────────────────────────────────┘        │
│                                                       │
│  容错：OneReason 结果不可用时 → 自动回退 OneRec       │
└─────────────────────────────────────────────────────┘
```

**四步工作流**：

| 步骤 | 名称 | 说明 |
|:---:|------|------|
| **1** | Pipeline Design | 解耦的近线检索管道，**不参与早期召回竞争**，在下游排序模型中与 OneRec 实时结果联合打分；OneReason 结果不可用时自动回退 |
| **2** | Dataset Trigger | **周期性**（每天/每小时）聚合分布式数据作业，构建用户行为、画像、item 内容数据，触发在线管道进行数据抓取、清洗和特征处理，形成端到端数据飞轮 |
| **3** | Offline Inference | 加载最新 OneReason checkpoint，**离线**预测用户最可能的 next itemic tokens，用专用解码策略解码为 item ID |
| **4** | Online Serving | 解码后的 item ID 写入 **Redis**，在线推理时与 OneRec 实时检索结果联合送入排序模型统一融合 |

---

### 11.3 两种应用范式

| 范式 | 管道 | 机制 | 效果 |
|------|:---:|------|------|
| **OneReason 直接检索** | Slow | 直接预测 next itemic tokens → 解码为候选 item 列表 | 提升流量**质量**（转化率/点击率显著提升） |
| **OneReason for OneRec** | Fast | 将 OneReason 输出**蒸馏**到 OneRec 中，引入 Thinking Token + Alignment Network | 提升系统**规模**（检索覆盖率大幅扩展） |

**OneReason for OneRec 蒸馏框架详解**：

1. OneReason 预测最高概率的 next itemic token
2. 通过量化模型解码为 embedding 表示，作为推理过程的监督信号
3. 在 OneRec 中引入 **Thinking Token**，使用 decoder 侧 `<BOS>` 位置的 hidden state 作为监督目标
4. 通过 **Alignment Network** 施加表示级约束，在语义空间中蒸馏 OneReason 的推理能力
5. 扩展设计：User Representation Decoder 压缩编码器输出 + 双分支架构（Decoder with/without Reason）+ 基于空间残差的对比蒸馏方法
