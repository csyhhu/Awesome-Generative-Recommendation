## 1. 研究背景与动机

美团作为领先的生活服务电商平台，2024年服务了超过7.7亿交易用户，日峰值处理超9800万订单。推荐系统是其核心业务组件。

**生成式推荐模型 (GRM)** 相比传统深度推荐模型 (DRM) 具有更高的准确率，因为它能够通过自注意力机制建模用户完整的行为序列（而非仅建模单用户-单物品对）。然而，GRM 的计算复杂度随序列长度呈平方增长（相对于 DRM 的线性增长），使得高效训练面临巨大挑战。

现有系统（以 Meta 的 TorchRec 为代表）存在两个关键局限：

1. **静态 Embedding 表无法处理动态变化**：无法实时为新用户/新商品（如商家更新菜单）分配 embedding 向量，固定大小的静态表需要过度预分配容量，导致内存浪费
2. **Embedding 查找效率低下**：用户序列中存在大量重复特征 ID，导致重复传输 embedding 向量；缺少自动表合并机制，需要大量人工配置

## 2. MTGenRec 核心设计

MTGenRec 基于 TorchRec 开发，采用**稀疏模型（模型并行）+ 稠密模型（数据并行）** 的混合并行策略，端到端同步训练。

### 2.1 动态 Embedding 表（Dynamic Embedding Table）

采用**基于哈希的解耦存储架构**：

- **Key-Value 分离存储**：轻量级 Key 结构维护键到 embedding 指针的映射表，Value 结构存储 embedding 向量和辅助数据（时间戳等）。扩容时优先扩展较小的 Key 结构，避免大量 embedding 数据搬运
- **基于 chunk 的嵌入分配**：多个 embedding 存放在一个 chunk 中，按 chunk 粒度进行插入/删除，减少单个操作次数
- **哈希函数**：选用 MurmurHash3，其对输入 ID 按 4 字节块执行混合操作，最大化熵值和雪崩效应
- **碰撞处理 —— 分组并行探测（Grouped Parallel Probing）**：
  - 确保基础步长 S 为奇数（通过位或操作），保证能遍历表中所有位置
  - 将步长与键值关联，避免不同键的探测序列重叠导致聚集
  - 将步长乘以线程组编号，使不同线程组探测表的不同区域
  - 数学公式：$S = (k\%(M/\text{threads}-1)+1 \mid 1) \times \text{threads}$
  - 通过互质条件证明了该技术可以遍历哈希表中所有槽位
- **容量扩展**：当负载因子 > 0.75 时触发扩容，容量以 2 的幂次倍增；Key 结构优先扩容，同时维护当前 chunk 和下一个 chunk 的双 chunk 机制确保持续可写入

### 2.2 自动 Embedding 表合并（Automated Table Merging）

- 设计统一的 `FeatureConfig` 配置接口，为每个特征定义名称、embedding 维度、查找表等参数
- 系统自动生成合并策略（如合并相同维度的表）
- 引入 `HashTableCollection` 自动构建哈希表集合，开发者只需指定所需特征，无需手动干预
- 对于动态表的 ID 冲突问题：使用位运算动态计算偏移量——使用高位作为特征标识位，确保全局唯一 ID
  - 计算公式：$\text{ID} = (i \ll (63 - k)) \mid x$，其中 $k = \lceil \log_2(m+1) \rceil$

### 2.3 两阶段 ID 去重（Two-stage ID Deduplication）

- **第一阶段**：在设备间交换 ID 前，先在本地去重，减少 ID 通信量
- **第二阶段**：ID 交换后，不同设备可能发送重复 ID，再次去重
- **核心目标**：减少 embedding 向量通信延迟（ID 通信影响相对较小）
- 实验显示吞吐量提升 1.1–3.7 倍

### 2.4 动态序列批处理（Dynamic Sequence Batching）

针对用户序列长度呈长尾分布导致的 GPU 负载不均问题：

- 每个 GPU 维护序列缓冲区，从 Hive 表块获取数据
- 定义目标 token 数 = 平均序列长度 × batch size
- 计算缓冲区中样本 token 数的累积和，通过二分搜索找到最接近目标 token 数的分割点
- 在数据并行中，通过 All-to-All 通信同步各设备 batch size，按 batch size 比例进行梯度加权平均，确保数值正确性
- 优势：① 通过 token 约束实现负载均衡；② 自适应 batch size 最大化 GPU 内存利用率

### 2.5 其他优化

- **流水线技术**：使用 copy/dispatch/compute 三个异步流重叠 I/O、表查找和计算
- **检查点恢复**：每设备独立保存 checkpoint，新设备通过取模运算定位所需文件
- **混合精度训练**：稠密模型 FP32→FP16；嵌入表按访问频率分区（热特征保留 FP32，冷特征使用 FP16）
- **梯度累积**：跨 batch 累积梯度，稀疏参数仅更新已激活部分
- **算子融合**：为 HSTU 模块定制 SRAM 内分块计算，使用 casual mask 跳过无效计算

## 3. 模型架构

Meituan 的 GRM 由四个核心组件构成：

1. **输入序列**：上下文序列、历史行为序列（点击/购买）、曝光序列
2. **Embedding 层**：将离散特征 ID 转换为连续 embedding 向量
3. **HSTU 层**：基于自注意力的 Transformer 架构，包含 U（用户特征提取）、Q/K/V、O（物品特征提取）、H（特征交互）四个注意力子层
4. **MMoE 层**：多门控混合专家网络，动态平衡多目标任务

## 4. 实验与结果

### 4.1 实验设置

- **模型规模**：Small GRM (4G FLOPs) 和 Large GRM (110G FLOPs)，embedding 维度因子 1D/8D/64D
- **数据**：美团 10 天用户日志，每天超 2 亿序列，平均序列长度 600，最大 3000
- **硬件**：NVIDIA A100 SXM4 80GB × 8 节点，最多扩展至 128 GPUs
- **评估指标**：GAUC（准确率）、吞吐量（效率）

### 4.2 主要结果

| 指标 | 结果 |
|------|------|
| **吞吐量提升** | 相比 TorchRec 提升 1.6× – 2.44× |
| **可扩展性** | 128 GPU 时达到理想线性加速的 62.75%–78.5% |
| **序列均衡效果** | GRM 110G 平均吞吐提升 26.5%，64 GPU 时峰值 33.5% |
| **ID 去重效果** | 吞吐提升 1.1–3.7×（embedding 维度越大提升越显著） |
| **动态 Embedding 表 vs MCH** | 吞吐提升 1.47–2.22×，且避免 OOM |
| **在线 A/B 测试** | 外卖订单量 +1.22%，PV_CTR +1.31%（GRM 110G vs 已训练 2 年的 DLRM） |

### 4.3 离线精度对比

| 模型 | CTR GAUC 提升 | CTCVR GAUC 提升 |
|------|-------------|---------------|
| GRM 4G vs DRM | +0.0036 | +0.0154 |
| GRM 110G vs DRM | +0.0153 | +0.0288 |

## 5. 贡献总结

1. 提出了 **MTGenRec**，一个面向美团工业级 GRM 训练的端到端高效分布式系统
2. 针对稀疏模型：提出**动态嵌入表**处理实时 ID 变化、**两阶段 ID 去重**和**自动表合并**加速 embedding 查找
3. 针对稠密模型：提出**动态序列批处理**解决 GPU 间负载不均
4. 系统已在美团多个业务线部署，日均处理数亿请求

## 6. 论文亮点与思考

- **工业落地性强**：论文从美团实际业务痛点出发（新商家上线、序列长度不均等），提出的每项优化都有明确的工程动机
- **轻量化设计理念突出**：动态序列均衡部分提到，虽然有更复杂的序列打包或动态并行策略，但在当前数据条件下，轻量级方案已接近最优，无需引入额外复杂度
- **Hash 表碰撞处理的数学严谨性**：对分组并行探测技术给出了完整的数学证明（基于互质条件和鸽子洞原理）
- **系统层面全面**：除核心算法优化外，还涵盖了流水线、检查点恢复、混合精度、梯度累积、算子融合等工程实现细节
