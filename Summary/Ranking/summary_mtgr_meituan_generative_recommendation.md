# MTGR：美团工业级生成式推荐框架 — 阅读笔记

- **论文**：MTGR: Industrial-Scale Generative Recommendation Framework in Meituan  
- **arXiv**：<https://arxiv.org/abs/2505.18654>  
- **会议**：CIKM 2025（文中 ACM DOI：10.1145/3746252.3761565）

---

## 1. 背景与动机

缩放定律在 NLP、CV、检索等领域已被广泛验证。推荐排序里有两条主流路线：**DLRM（深度学习推荐模型）**与**生成式推荐（GRM）**。DLRM 依赖大量手工特征（尤其是 **交叉特征**，刻画用户与候选之间的交互统计），效果好，但面临两点瓶颈：（1）用户行为爆炸式增长时，难以高效吃全长序列，往往退化为检索序列或低复杂度模块；（2）训练与推理代价大致随候选数线性增长，成本高。

GRM（如基于 Transformer/HSTU 的方法）把行为链当作 token 序列建模，能压缩同一用户下的冗余样本、用高效注意力做端到端长序列编码，扩展性强；但典型 **下一 token 预测** 范式难以在「还不知道具体候选」的情况下使用 **用户–候选交叉特征**。作者发现：**去掉交叉特征会显著伤效果，且单纯放大模型也无法弥补**。

**核心问题**：如何在保留 DLRM 级别特征（含交叉特征）的前提下，获得接近 GRM 的可扩展性？

---

## 2. MTGR 思路概览

**MTGR（Meituan Generative Recommendation）** 在建模骨干上采用与 Actions Speak Louder（HSTU）类似的 **encoder-only 自注意力栈**，但通过 **数据重排 + 判别式损失 + 候选 token 显式携带交叉特征**，把 DLRM 与 GRM 的优点结合起来：

1. **用户侧样本聚合（训练按窗口、推理按请求）**  
   传统对每个 `(用户, 候选 i)` 单独前向；MTGR 将同一用户的 \(K\) 个候选聚合成一条样本  
   \(\mathbb{D} = [\mathbf{U}, \overrightarrow{\mathbf{S}}, \mathbf{R}_t, [\mathbf{C},\mathbf{I}]_1,\ldots,[\mathbf{C},\mathbf{I}]_K]\)，  
   其中 \(\mathbf{C}\) 为交叉特征，与候选 \(\mathbf{I}\) 一并组成 **候选 token**。这样 **对用户表征只做一次计算**，同时为所有候选打分，训练样本量从「曝光×候选」量级降到「用户×聚合窗口」量级，推理上对单次请求的多候选 **只做一次推理**，候选规模对耗费的弹性优于传统 DLRM 的「每候选一遍」。

2. **Token 化**  
   用户 profile 的每个标量特征一个 token；历史序列与实时序列里每个 item 一个 token；每个候选（交叉+物品特征）一个 token，经嵌入与 MLP 统一到同一维度 \(d_{\text{uni}}\)。

3. **统一 HSTU 编码器**  
   每层：Group Layer Norm → Q/K/V/U 投影 → 带 SiLU 的注意力（注意力权重除以序列总长度作均值因子）→ 与 \(\mathbf{U}\) 点乘更新 → 再 GroupLN、残差与 MLP。**未使用传统 FFN**（与部分缩放工作一致）。

4. **Group-Layer Normalization（GLN）**  
   按 **特征域分组**（用户、历史序列、实时序列、候选等）分别归一化，缓解异构语义空间混在同一序列里时的分布不匹配问题。

5. **动态掩码（Dynamic Masking）**  
   简单因果掩码在「训练聚合窗口内混入不同时刻请求」时会造成 **信息泄漏**（例如晚间行为不应影响下午候选）。规则概括为：  
   - 用户与长历史（静态序列）对所有 token **全可见**；  
   - 实时序列 \(\mathbf{R}_t\) **按真实时间因果**：早发生的 token 看不到晚发生的；  
   - **候选 token 之间互相隔离**（仅见自身），避免候选间相互「泄题」。  

6. **训练系统**  
   从 TensorFlow 迁到 PyTorch/TorchRec，并针对工业流式训练做了：**动态哈希稀疏表**（解耦 key/value 存储、可扩容）、**嵌入查询去重与通信优化**、**动态 batch size 做序列长度负载均衡**（按实际长度调各卡 BS，梯度按 BS 加权）、**copy/dispatch/compute 三流流水线**、**bf16**、基于 CUTLASS 的注意力核等。相对原版 TorchRec 声称训练吞吐约 **1.6×–2.4×**，百卡扩展性良好。

## 3. 实验与结论要点

- **数据**：公开数据交叉特征弱；文在美团真实日志上构造数据（10 天离线、线上对比用 6 个月以上），强调交叉特征与长序列。
- **离线**：CTR / CTCVR 任务上报告 AUC、GAUC；最强 DLRM 基线为 UserTower-SIM 等。**MTGR-small 即超过最强 DLRM**；small / medium / large 随算力平滑提升。
- **消融**：去掉交叉特征、去掉 GLN、去掉动态掩码均显著掉点；去交叉特征的损失大到 **几乎抵消 large 相对 DLRM 的收益**，印证交叉特征在工业场景的关键作用。
- **可扩展性**：随层数、\(d_{\text{model}}\)、序列长度增加，性能平滑上升；文中给出相对最强 DLRM 的收益与算力倍数之间的 **幂律** 关系示意图。
- **线上**：外卖场景约 **2%** 流量 AB；相对持续迭代两年的先进 DLRM（UserTower-SIM），large 在文中表格体现离线 CTR GAUC +0.0153、CTCVR GAUC +0.0288，线上 PV_CTR +1.90%、UV_CTCVR +1.02% 等（具体数值以论文表为准）。**已全量部署主流量**；相对 DLRM：**训练成本基本持平**，前向算力约 **65× FLOPs/样本**，但借助聚合推理 **推理成本约降 12%**。

---

## 4. 个人小结

MTGR 的实质是：**用「用户维度的样本聚合 + token 序列 + HSTU」拿到 GRM 式的计算与序列伸缩红利**，同时把 **交叉特征绑在候选 token 上** 并用 **判别式打分** 保留 DLRM 的信息优势；再用 **GLN** 与 **动态掩码** 处理异构语义与因果/泄漏。工程侧通过 TorchRec 上一套稀疏与并行优化，使「远高于 DLRM 单样本算力」的大模型在真实训练资源约束下可落地。文末展望往 **多场景统一、推荐基础模型** 方向延伸。
